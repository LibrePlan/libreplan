Конфігурація LDAP
##################

.. contents::

Цей екран дозволяє встановити з'єднання з LDAP для делегування
автентифікації та/або авторизації.

Він розділений на чотири різні області, які пояснюються нижче:

Активація
==========

Ця область використовується для встановлення властивостей, що визначають, як *LibrePlan* використовує
LDAP.

Якщо поле *Увімкнути автентифікацію LDAP* позначено, *LibrePlan* звертатиметься до
LDAP щоразу, коли користувач намагається увійти до застосунку.

Позначене поле *Використовувати ролі LDAP* означає, що встановлюється відповідність між ролями LDAP та
ролями LibrePlan. Відповідно, дозволи для користувача в
LibrePlan залежатимуть від ролей користувача в LDAP.

Конфігурація
=============

Цей розділ містить значення параметрів для доступу до LDAP. *Base*, *UserDN* та
*Password* — це параметри, що використовуються для підключення до LDAP та пошуку користувачів. Тому
вказаний користувач повинен мати дозвіл на виконання цієї операції в LDAP. Внизу цього
розділу є кнопка для перевірки можливості з'єднання з LDAP із заданими параметрами.
Рекомендується перевірити з'єднання перед тим, як продовжувати конфігурацію.

.. NOTE::

   Якщо ваш LDAP налаштовано для роботи з анонімною автентифікацією, можна
   залишити атрибути *UserDN* та *Password* порожніми.

.. TIP::

   Щодо конфігурації *Active Directory (AD)*, поле *Base* повинно бути
   точним місцезнаходженням підключеного користувача в AD.

   Приклад: ``ou=organizational_unit,dc=example,dc=org``

Автентифікація
===============

Тут можна налаштувати властивість у вузлах LDAP, де має знаходитися вказане ім'я користувача.
Властивість *UserId* має бути заповнена назвою властивості, де зберігається ім'я користувача в LDAP.

Прапорець *Зберігати паролі в базі даних*, якщо позначений, означає, що
пароль також зберігається в базі даних LibrePlan. Таким чином, якщо LDAP недоступний
або недосяжний, користувачі LDAP можуть автентифікуватися через базу даних
LibrePlan. Якщо не позначений, користувачі LDAP можуть автентифікуватися лише через
LDAP.

Авторизація
============

Цей розділ дозволяє визначити стратегію відповідності ролей LDAP ролям
LibrePlan. Перший вибір — це стратегія залежно від
реалізації LDAP.

Стратегія груп
--------------

Коли ця стратегія використовується, вона вказує, що LDAP має стратегію ролей-груп.
Це означає, що користувачі в LDAP є вузлами, що знаходяться безпосередньо під гілкою, яка
представляє групу.

Наступний приклад представляє дійсну структуру LDAP для використання стратегії груп.

* Структура LDAP::

   dc=example,dc=org
   |- ou=groups
      |- cn=admins
      |- cn=itpeople
      |- cn=workers
   |- ou=people
      |- uid=admin1
      |- uid=it1
      |- uid=it2
      |- uid=worker1
      |- uid=worker2
      |- uid=worker3

У цьому випадку кожна група матиме атрибут, наприклад ``member``,
зі списком користувачів, що належать до групи:

* ``cn=admins``:

  * ``member: uid=admin1,ou=people,dc=example,dc=org``
  * ``member: uid=it1,ou=people,dc=example,dc=org``

* ``cn=itpeople``:

  * ``member: uid=it1,ou=people,dc=example,dc=org``
  * ``member: uid=it2,ou=people,dc=example,dc=org``

* ``cn=workers``:

  * ``member: uid=worker1,ou=people,dc=example,dc=org``
  * ``member: uid=worker2,ou=people,dc=example,dc=org``
  * ``member: uid=worker3,ou=people,dc=example,dc=org``

Конфігурація для цього випадку така:

* Стратегія пошуку ролей: ``Group strategy``
* Шлях до групи: ``ou=groups``
* Властивість ролі: ``member``
* Запит пошуку ролі: ``uid=[USER_ID],ou=people,dc=example,dc=org``

І, наприклад, для відповідності ролей:

* Адміністрування: ``cn=admins;cn=itpeople``
* Читач вебсервісів: ``cn=itpeople``
* Записувач вебсервісів: ``cn=itpeople``
* Дозволено читання всіх проєктів: ``cn=admins``
* Дозволено редагування всіх проєктів: ``cn=admins``
* Дозволено створення проєктів: ``cn=workers``

Стратегія властивостей
-----------------------

Коли адміністратор вирішує використовувати цю стратегію, це вказує, що кожен
користувач є вузлом LDAP, і всередині вузла існує властивість, що представляє
групу(и) для користувача. У цьому випадку конфігурація не вимагає параметра *Group path*.

Наступний приклад представляє дійсну структуру LDAP для використання стратегії властивостей.

* Структура LDAP::

   dc=example,dc=org
   |- ou=people
      |- uid=admin1
      |- uid=it1
      |- uid=it2
      |- uid=worker1
      |- uid=worker2
      |- uid=worker3

**За атрибутом**

У цьому випадку кожен користувач матиме атрибут, наприклад ``group``,
з назвою групи, до якої вони належать:

* ``uid=admin1``:

  * ``group: admins``

* ``uid=it1``:

  * ``group: itpeople``

* ``uid=it2``:

  * ``group: itpeople``

* ``uid=worker1``:

  * ``group: workers``

* ``uid=worker2``:

  * ``group: workers``

* ``uid=worker3``:

  * ``group: workers``


.. WARNING::

   Ця стратегія має обмеження: кожен користувач може належати лише до однієї групи.

Конфігурація для цього випадку така:

* Стратегія пошуку ролей: ``Property strategy``
* Шлях до групи:
* Властивість ролі: ``group``
* Запит пошуку ролі: ``[USER_ID]``

І, наприклад, для відповідності ролей:

* Адміністрування: ``admins;itpeople``
* Читач вебсервісів: ``itpeople``
* Записувач вебсервісів: ``itpeople``
* Дозволено читання всіх проєктів: ``admins``
* Дозволено редагування всіх проєктів: ``admins``
* Дозволено створення проєктів: ``workers``

**За ідентифікатором користувача**

Можна навіть використати обхідний шлях для призначення ролей LibrePlan безпосередньо
користувачам без наявності атрибута в кожному користувачі LDAP.

У цьому випадку ви вказуєте, які користувачі мають різні ролі LibrePlan
за ``uid``.

Конфігурація для цього випадку така:

* Стратегія пошуку ролей: ``Property strategy``
* Шлях до групи:
* Властивість ролі: ``uid``
* Запит пошуку ролі: ``[USER_ID]``

І, наприклад, для відповідності ролей:

* Адміністрування: ``admin1;it1``
* Читач вебсервісів: ``it1;it2``
* Записувач вебсервісів: ``it1;it2``
* Дозволено читання всіх проєктів: ``admin1``
* Дозволено редагування всіх проєктів: ``admin1``
* Дозволено створення проєктів: ``worker1;worker2;worker3``

Відповідність ролей
--------------------

Внизу цього розділу є таблиця з усіма ролями LibrePlan
та текстовим полем поруч з кожною. Це призначено для відповідності ролей. Наприклад,
якщо адміністратор вирішить, що роль *Адміністрування* LibrePlan відповідає
ролям *admin* та *administrators* у LDAP, текстове поле повинно містити:
"``admin;administrators``". Символ для розділення ролей — "``;``".

.. NOTE::

   Якщо хочете вказати, що всі користувачі або всі групи мають один дозвіл, можна
   використовувати зірочку (``*``) як символ підстановки для посилання на них. Наприклад, якщо
   хочете, щоб усі мали роль *Дозволено створення проєктів*, налаштуйте
   відповідність ролей так:

   * Дозволено створення проєктів: ``*``
