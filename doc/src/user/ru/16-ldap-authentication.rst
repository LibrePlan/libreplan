Настройка LDAP
##############

.. contents::

Этот экран позволяет установить соединение с LDAP для делегирования
аутентификации и/или авторизации.

Он разделён на четыре различные области, которые описаны ниже:

Активация
=========

Эта область используется для настройки свойств, определяющих, как *LibrePlan* использует
LDAP.

Если поле *Включить LDAP-аутентификацию* отмечено, *LibrePlan* будет запрашивать
LDAP каждый раз, когда пользователь пытается войти в приложение.

Поле *Использовать роли LDAP* при включении означает, что устанавливается соответствие между ролями LDAP и
ролями LibrePlan. Следовательно, права пользователя в
LibrePlan будут зависеть от ролей, которые пользователь имеет в LDAP.

Конфигурация
============

В этом разделе содержатся значения параметров для доступа к LDAP. *Base*, *UserDN* и
*Password* — это параметры, используемые для подключения к LDAP и поиска пользователей. Поэтому
указанный пользователь должен иметь разрешение на выполнение этой операции в LDAP. В
нижней части этого раздела есть кнопка для проверки возможности соединения с LDAP с
заданными параметрами. Рекомендуется проверить соединение перед
продолжением настройки.

.. NOTE::

   Если ваш LDAP настроен для работы с анонимной аутентификацией, можно
   оставить атрибуты *UserDN* и *Password* пустыми.

.. TIP::

   Относительно конфигурации *Active Directory (AD)*: поле *Base* должно быть
   точным расположением привязанного пользователя в AD.

   Пример: ``ou=organizational_unit,dc=example,dc=org``

Аутентификация
==============

Здесь можно настроить свойство в узлах LDAP, где должно находиться имя пользователя.
Свойство *UserId* должно быть заполнено именем
свойства, в котором хранится имя пользователя в LDAP.

Флажок *Сохранять пароли в базе данных* при включении означает, что
пароль также хранится в базе данных LibrePlan. Таким образом, если LDAP
недоступен, пользователи LDAP могут аутентифицироваться через базу данных
LibrePlan. Если флажок не установлен, пользователи LDAP могут аутентифицироваться только через
LDAP.

Авторизация
===========

Этот раздел позволяет определить стратегию сопоставления ролей LDAP с
ролями LibrePlan. Первый выбор — это стратегия, которую нужно использовать в зависимости от
реализации LDAP.

Стратегия групп
---------------

При использовании этой стратегии указывается, что LDAP имеет стратегию ролей-групп.
Это означает, что пользователи в LDAP — это узлы, непосредственно находящиеся под ветвью,
представляющей группу.

Следующий пример представляет допустимую структуру LDAP для использования стратегии групп.

* Структура LDAP::

   dc=example,dc=org
   |- ou=groups
      |- cn=admins
      |- cn=itpeople
      |- cn=workers
   |- ou=people
      |- uid=admin1
      |- uid=it1
      |- uid=it2
      |- uid=worker1
      |- uid=worker2
      |- uid=worker3

В этом случае каждая группа будет иметь атрибут, например называемый ``member``,
со списком пользователей, принадлежащих группе:

* ``cn=admins``:

  * ``member: uid=admin1,ou=people,dc=example,dc=org``
  * ``member: uid=it1,ou=people,dc=example,dc=org``

* ``cn=itpeople``:

  * ``member: uid=it1,ou=people,dc=example,dc=org``
  * ``member: uid=it2,ou=people,dc=example,dc=org``

* ``cn=workers``:

  * ``member: uid=worker1,ou=people,dc=example,dc=org``
  * ``member: uid=worker2,ou=people,dc=example,dc=org``
  * ``member: uid=worker3,ou=people,dc=example,dc=org``

Конфигурация для этого случая следующая:

* Стратегия поиска ролей: ``Group strategy``
* Путь к группе: ``ou=groups``
* Свойство роли: ``member``
* Запрос поиска роли: ``uid=[USER_ID],ou=people,dc=example,dc=org``

И, например, если нужно сопоставить некоторые роли:

* Administration: ``cn=admins;cn=itpeople``
* Web service reader: ``cn=itpeople``
* Web service writer: ``cn=itpeople``
* All projects read allowed: ``cn=admins``
* All projects edition allowed: ``cn=admins``
* Project creation allowed: ``cn=workers``

Стратегия свойств
-----------------

Когда администратор решает использовать эту стратегию, это означает, что каждый пользователь
является узлом LDAP, и внутри узла существует свойство, представляющее
группу(ы) пользователя. В этом случае конфигурация не требует
параметра *Group path*.

Следующий пример представляет допустимую структуру LDAP для использования стратегии свойств.

* Структура LDAP::

   dc=example,dc=org
   |- ou=people
      |- uid=admin1
      |- uid=it1
      |- uid=it2
      |- uid=worker1
      |- uid=worker2
      |- uid=worker3

**С атрибутом**

В этом случае каждый пользователь будет иметь атрибут, например называемый ``group``,
с именем группы, к которой они принадлежат:

* ``uid=admin1``:

  * ``group: admins``

* ``uid=it1``:

  * ``group: itpeople``

* ``uid=it2``:

  * ``group: itpeople``

* ``uid=worker1``:

  * ``group: workers``

* ``uid=worker2``:

  * ``group: workers``

* ``uid=worker3``:

  * ``group: workers``


.. WARNING::

   Эта стратегия имеет ограничение: каждый пользователь может принадлежать только одной группе.

Конфигурация для этого случая следующая:

* Стратегия поиска ролей: ``Property strategy``
* Путь к группе:
* Свойство роли: ``group``
* Запрос поиска роли: ``[USER_ID]``

И, например, если нужно сопоставить некоторые роли:

* Administration: ``admins;itpeople``
* Web service reader: ``itpeople``
* Web service writer: ``itpeople``
* All projects read allowed: ``admins``
* All projects edition allowed: ``admins``
* Project creation allowed: ``workers``

**По идентификатору пользователя**

Можно также использовать обходной приём для назначения ролей LibrePlan непосредственно пользователям
без наличия атрибута в каждом пользователе LDAP.

В этом случае указывается, какие пользователи имеют различные роли LibrePlan,
по ``uid``.

Конфигурация для этого случая следующая:

* Стратегия поиска ролей: ``Property strategy``
* Путь к группе:
* Свойство роли: ``uid``
* Запрос поиска роли: ``[USER_ID]``

И, например, если нужно сопоставить некоторые роли:

* Administration: ``admin1;it1``
* Web service reader: ``it1;it2``
* Web service writer: ``it1;it2``
* All projects read allowed: ``admin1``
* All projects edition allowed: ``admin1``
* Project creation allowed: ``worker1;worker2;worker3``

Сопоставление ролей
-------------------

В нижней части этого раздела находится таблица со всеми ролями LibrePlan
и текстовым полем рядом с каждой из них. Это используется для сопоставления ролей. Например,
если администратор решает, что роль LibrePlan *Administration* соответствует
ролям *admin* и *administrators* в LDAP, текстовое поле должно содержать:
"``admin;administrators``". Символ для разделения ролей — "``;``".

.. NOTE::

   Если вы хотите указать, что все пользователи или все группы имеют одно разрешение, можно
   использовать звёздочку (``*``) в качестве подстановочного знака для ссылки на них. Например, если
   нужно, чтобы у всех было право *Project creation allowed*, настройте сопоставление ролей следующим образом:

   * Project creation allowed: ``*``
